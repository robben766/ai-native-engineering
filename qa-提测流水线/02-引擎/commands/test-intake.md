提测验证流水线：研发提测后，自动完成变更分析→用例生成→代码review→门禁→部署闸口→E2E/接口验证，并回写工作项系统。QA 专用。

用法：
```
/test-intake MR=<MR链接 或 仓库+分支>  ISSUE=<工作项号>  [DOC=<设计文档路径/链接>]  [ENV=<url> TENANT=<租户> ACCOUNT=<账号>]
```
参数写法灵活，用户没按 key=value 给也没关系，从 `$ARGUMENTS` 里识别 MR/分支、工作项号、环境信息即可。

---

## 全局边界（铁律，全程遵守）
- 只读代码 + 评论工作项系统 + 写用例（tcase）。**绝不提交业务代码、绝不动任何主线分支、绝不 git commit/push。**
- 本流水线文件在知识库仓，**未跑通前不 commit/push**。经验文档 append 也只改本地。
- 仅面向主线，不涉及特定客户的定制分支。
- 工作项系统 / 代码平台 / Playwright MCP **在主会话直接调用**。（子 Agent MCP 可用性以实测为准；不确定就留主会话。）

## 设计与经验层
- 总设计：`docs/test-intake-pipeline.md`
- 经验层：`docs/experience/`（platform / review-rules / e2e-patterns / projects/<业务仓>）——**每阶段按目标仓自动加载**。

---

## 执行流程

### 阶段 0 — 入参归一 + 判仓 + 加载经验
1. 从 `$ARGUMENTS` 解析：MR链接/仓库+分支、工作项号、DOC、ENV/TENANT/ACCOUNT。缺 MR 或工作项号则向用户索取。
2. **判目标仓**：给了仓库名直接用；否则先做阶段①拿到改动后按「<领域文档>判仓规则」双证据判定（注意存在同名模块与独立仓歧义时，不能仅凭包名/类名判仓，须用控制器实际文件路径 + API 路由前缀双证据）。
3. **加载经验**：`docs/experience/platform.md` + `docs/experience/projects/<目标仓>.md`（不存在则用 `_template.md` 起一个）。

### 阶段 1 — 需求解析（内联，工作项系统 MCP 直调）
- 按 story→bug→task 顺序查工作项系统 `$工作项号`，拿标题/描述/验收标准/复现步骤/附件截图。
- 输出「需求要点」，作为后续阶段输入。

### 阶段 2 — 变更提取
- 调用 skill **`mr-change-extract`**，产出改动清单 + 影响面 + 回归范围。
- 若阶段0未判仓，此处确定目标仓并回填经验加载。

### 阶段 3 — 用例生成
- 调用 skill **`testcase-generate`**，产出完整用例。
- **QA 确认后**：写入工作项系统用例库 + 关联工作项 + 发用例摘要评论。

### 阶段 4 — 代码 review
- 调用 skill **`code-review-gate`**，产出问题清单+风险分级+自动裁决。
- **QA 确认后**：发 review 评论到工作项系统。

### 阶段 5 — 门禁
- 读阶段4自动裁决：
  - 不通过 → **停**，输出待修复清单，提示"等研发修复后重跑 `/test-intake`"。结束。
  - 有风险 → 提示 QA，等 `覆盖放行` 或 `覆盖拦截`。
  - 通过 → 继续。
- 任何时候 QA 可手动覆盖裁决（记录理由）。

### 阶段 6 — 部署（人工闸口 / 未来可插拔）
- **首期**：提示 QA「门禁已通过，请部署到测试环境；部署完成后回复『就绪』并提供 ENV/TENANT/ACCOUNT（若阶段0未给）」。**挂起等待**用户确认，不自行部署。
- **未来**：此处替换为「调用运维平台 MCP 触发部署 + 轮询就绪」，前后阶段不变。

### 阶段 7 — E2E / 接口验证
- 确认环境可达（本会话能访问 ENV）。
- 调用 skill **`e2e-verify`**：执行可自动化用例 + 接口断言 + 截图。
- 发执行结果评论到工作项系统。

### 阶段 8 — 汇总
- 汇总 review 结论 + 用例覆盖 + E2E 结果为一份总报告，发工作项系统汇总评论（QA 确认后）。
- 询问是否有需沉淀的经验；若有，append 到对应 `docs/experience/` 文件（只改本地）。

### 阶段 8.5 — 度量落库（自动，不阻断）
- 调用 skill **`qa-metrics`**：把本次运行的过程指标（用例数/维度/可自动化、review 各级别数+去误报、门禁裁决、E2E 结果、耗时）汇总成一条记录，追加到 `docs/metrics/intake-runs.jsonl`。
- **失败仅告警，绝不影响提测结论**。这一步只写度量文件，不碰业务代码、不发工作项系统。
- 先把尺子造出来，后续每个改进才有得量。结果指标（逃逸率/一次通过率）需接工作项数据，见 `docs/metrics/README.md`。

---

## 分阶段可停
用户可只跑到某阶段（如"只做 review"）或从某阶段起跑；各 skill 均可单独调用。默认全流程，遇门禁不通过或部署闸口自然暂停。
