# 提测验证流水线（Codex 适配版）

本项目启用「提测验证流水线」。研发提测后，按下面的流程完成变更分析 → 用例生成 → 代码评审 → 门禁 → 部署闸口 → E2E 验证，并回写工作项系统。QA 专用。

> 若本仓已有其他 AGENTS.md 内容，把本段合并进去即可。

## 全局边界（铁律，全程遵守）

- 只做三件事：**读代码、评论工作项系统、写测试用例**。
- **绝不提交业务代码、绝不动任何主线分支、绝不 git commit / push。**
- 仅面向主线，不涉及特定客户的定制分支。
- 工作项系统 / 代码平台 / 浏览器（Playwright）**通过 MCP 调用**（配置见 `config.toml`）。

## 触发

当我说「跑提测验证 / test-intake」并给出 **MR/分支 + 工作项号**（可选 ENV/TENANT/ACCOUNT）时，按下面七环**顺序执行**，遇到标注「⏸ 等我确认」的地方**停下等我**，不要一口气跑完。

参数写法灵活，从我的话里识别 MR/分支、工作项号、环境信息即可；缺 MR 或工作项号就问我。

## 经验层（每阶段按目标仓自动加载）

先读 `docs/experience/platform.md` + `docs/experience/projects/<目标仓>.md`；评审阶段读 `review-rules.md`、判准读 `oracles.md`。这些是判断依据，不是可选参考。

## 执行流程（七环，串行）

**阶段 0 · 入参归一 + 判仓 + 加载经验**
解析 MR/分支、工作项号、环境信息；确定目标仓（给了仓名直接用，否则先取改动再按仓库判定规则用「控制器文件路径 + API 路由前缀」双证据判仓）；加载对应经验层。

**阶段 1 · 需求解析**
用工作项系统 MCP 查工作项号，拿标题/描述/验收标准/复现步骤/附件，输出「需求要点」。

**阶段 2 · 变更提取**
加载 skill **`mr-change-extract`**：产出改动清单 + 影响面 + 回归范围。

**阶段 3 · 用例生成** ⏸ 等我确认
加载 skill **`testcase-generate`**：产出多维度用例（功能/异常/边界/权限/多租户/回归），消费 `oracles.md` 定预期。**我确认后**才写入工作项系统用例库 + 关联 + 发摘要评论。

**阶段 4 · 代码评审**
加载 skill **`code-review-gate`**。⚠️ Codex 无并行 workflow，改为**串行多维评审**：依次审「功能正确性 → 权限/多租户 → 边界/异常」三个维度，对照 `review-rules.md` 白盒审；**每条发现自我复核一次去误报**；产出问题清单 + 风险分级 + 门禁自动裁决。

**阶段 5 · 门禁**
按 `review-rules.md` 已签核的严重度基线裁决：
- **Blocker > 0 → 不通过**：停，输出待修清单，提示「研发修完后重跑」。结束。
- **仅 Major > 0 → 有风险** ⏸ 等我确认：默认不放行，我可 `覆盖放行` 或 `覆盖拦截`（记理由）。
- **无 Blocker/Major → 通过**：继续。

**阶段 6 · 部署闸口** ⏸ 等我确认
提示我「门禁已过，请部署到测试环境；部署完回复『就绪』并给 ENV/TENANT/ACCOUNT」。**挂起等待**，不自行部署。

**阶段 7 · E2E / 接口验证**
确认环境可达后，加载 skill **`e2e-verify`**：跑可自动化用例 + 接口断言 + 截图留证，发结果评论到工作项系统。

**阶段 8 · 汇总 + 度量落库** ⏸ 回写前确认
汇总评审结论 + 用例覆盖 + E2E 结果为总报告，我确认后发工作项系统汇总评论；加载 skill **`qa-metrics`** 追加一条度量记录到 `docs/metrics/intake-runs.jsonl`（失败仅告警，不阻断）；问我是否有要沉淀的经验，有则 append 到对应 `docs/experience/` 文件（只改本地）。

## 分阶段可停

我可以只跑到某阶段（如「只做 review」）或从某阶段起跑；各 skill 也可单独触发。默认全流程，遇门禁不通过或部署闸口自然暂停。
