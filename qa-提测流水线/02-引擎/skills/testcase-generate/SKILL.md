# Testcase Generate（用例生成）

提测验证流水线阶段③。产出**完整**测试用例（功能/异常/边界/权限/多租户/回归），不只是冒烟。

## 输入
- 工作项系统的需求/缺陷要点（阶段①已解析）。
- 变更提取结果（阶段②，`mr-change-extract` 输出）。
- （可选）研发设计文档。

## 前置边界
- 只读代码 + 写工作项系统用例。绝不改业务代码。

## 工作流

1. **加载经验**：`docs/experience/platform.md` + `docs/experience/projects/<业务仓>.md` + `docs/experience/oracles.md`（该模块已沉淀的"什么算对"判准，用来定预期结果，别重猜）+ `docs/experience/production-bug-patterns.md`（生产逃逸高频模式，回归靶子）。

2. **提炼验收标准**：从工作项系统的需求/缺陷描述 + 设计文档，列出"期望行为"。缺陷类还要覆盖"原缺陷复现路径 + 修复后应有行为"。

3. **按维度生成用例**（结合变更点，逐条覆盖）：
   - **功能正常**：主流程、验收标准逐条。
   - **异常**：非法输入、缺参、服务异常、超时。
   - **边界**：空/极值/最大长度/分页边界/精度（金额 BigDecimal）。
   - **权限**：不同角色可见性与操作权限、越权拦截（见 platform.md #3）。
   - **多租户**：≥2 租户数据隔离、切租户正确性（platform.md #1）。
   - **回归**：阶段②影响面里列出的老功能。
   - **生产高频回归**：变更若命中生产逃逸高发的风险类目（金额/对账/外部回传/附件/签署等）（`production-bug-patterns.md` P1–P7），默认加挂对应高频用例集，别漏最常逃逸的靶子。
   - **计算一致性**：涉及金额/费率时与核心计算引擎预期核对（platform.md #2）。

3.5. **自我证伪（对抗式复核，去虚假覆盖）**：用例生成后，换一个"挑刺"视角逐条审自己刚写的用例——
   - ① **预期结果（oracle）真的对吗**？规格模糊处别把"看起来合理"当成"正确"；对照 `oracles.md`，拿不准的标注"预期待人确认"，不硬编一个假预期。
   - ② 这条是**有效增量、还是和别条重复**？合并同质用例，别用数量凑覆盖。
   - ③ 覆盖是否**维度虚高**（量大但维度重复）？把无效/存疑用例剔出正式清单，不制造虚假覆盖感。
   - （对应 `code-review-gate` 的对抗式复核，此处用在用例这条链——地图警告 AI 会产出"似是而非"的用例制造虚假安全，这一步就是堵它。）
   - 确认下来的新判准，收尾时 append 回 `oracles.md`（带出处）。

4. **每条用例结构**：`编号 | 用例标题 | 前置条件 | 步骤 | 预期结果 | 优先级(P0/P1/P2) | 类型(功能/异常/边界/权限/多租户/回归) | 是否可自动化`。
   - "是否可自动化"给阶段⑦ e2e-verify 用（排除支付/CA 签署/验证码等，见 e2e-patterns.md）。

5. **回写工作项系统**（主会话直接调工作项系统 MCP）：
   - 经 QA 确认后调用批量用例写入/更新接口，写入用例库。
   - 与对应 需求/缺陷 **关联**（用例-需求关联接口，视工作项系统能力）。
   - 同时按流水线配置在对应工作项发**评论**：附用例清单摘要。

## 输出格式

```
## 测试用例（工作项 <id> / <仓>@<分支>）
共 n 条：P0 x / P1 y / P2 z；可自动化 m 条

| # | 标题 | 前置 | 步骤 | 预期 | 优先级 | 类型 | 可自动化 |
|---|------|------|------|------|--------|------|---------|
```

## 输出契约
- 覆盖 6 类维度，不遗漏权限与多租户。
- 每条标注优先级、类型、是否可自动化。
- 写工作项系统用例库前需 QA 确认。
