# MR Change Extract（变更提取）

提测验证流水线阶段①。把研发提测的分支/MR 变成结构化的「改动清单 + 影响面」，供下游用例生成、代码 review、E2E 使用。

## 输入
- MR 链接（`<代码平台>/.../merge_requests/<id>`）**或** 仓库名 + 分支名。
- （可选）目标仓已知则直接用。

## 前置边界
- **只读**。绝不切分支、绝不 commit/push、绝不改主线。
- 明确目标环境（如主线与特定客户定制分支属不同环境，勿混用）。

## 工作流

1. **取 diff**（主会话直接调「代码平台」MCP，勿委托子 Agent 以免丢 MCP）：
   - 给了 MR：用「代码平台」MCP 的获取 MR、拉取 MR 变更接口拿变更文件与 diff。
   - 给了仓库+分支：本地已 clone 则 `git -C <仓> fetch && git -C <仓> diff origin/<主分支>...origin/<分支>`（主分支见仓库配置：各仓主分支可能不同）；未 clone 则用「代码平台」MCP 的 compare 接口。

2. **判目标仓**（若未知）：按「双证据」判定（控制器文件路径 + API 路由前缀），注意跨仓同名模块的歧义（独立仓库与另一仓内的同名模块可能并存，禁止仅凭包名/类名判仓）。

3. **归纳改动清单**：
   - 改动文件分组：后端 Controller/Service/Mapper、前端页面/组件、SQL/migrate、配置。
   - 提取**变更的 API 接口**（路由前缀 + 方法）与**前端页面/路由**。

4. **反查影响面**（关键，禁止用 find/grep 扫仓库替代）：
   - 用「<路由索引>」反查变更接口 → 关联的前端页面、被哪些调用方触达。
   - 用「<领域文档>」对应领域文档 → 业务流程上下游、跨仓联动（如某些模块会同时牵动多个业务仓）。
   - 加载经验层平台通用经验 + 目标仓专属经验（`docs/experience/platform.md` + `docs/experience/projects/<目标仓>.md`），标注该仓已知易错点。

5. **产出回归范围**：不只新功能，列出可能被间接影响的老功能。

6. **自我证伪（对抗式复核，防漏判影响面）**：影响面产出后，换一个"挑漏"视角复核自己——
   - ① **有没有漏掉被间接触达的模块**？重点查跨仓联动（对账类、配置类常牵动多仓）、共用 Service/Mapper、异步任务与缓存、定时任务。
   - ② 列出的每条影响**是否可追溯证实**（指得到具体调用方/路由/业务文档）？猜的标"待确认"，别当既定影响写。
   - ③ 宁可多列可疑项让人排除，也不静默漏掉——漏判影响面是回归遗漏的头号来源。
   - （对应 `code-review-gate` 的对抗式复核，此处用在影响面这条链。）

## 输出格式

```
## 变更提取（<仓>@<分支> / MR <id>）

### 目标仓判定
<仓> —— 证据：<控制器路径> + <路由前缀>

### 改动清单
- 后端接口：<方法> <路由>（<Controller.method>） …
- 前端页面：<路由/组件> …
- SQL/migrate：<有/无，说明>
- 配置：<有/无>

### 影响面 & 回归范围
| 变更点 | 关联前端/调用方 | 跨仓联动 | 回归建议 |
|-------|---------------|---------|---------|

### 该仓已知易错点（来自经验层）
- ...
```

## 输出契约
- 必含：目标仓判定+证据、改动接口/页面清单、影响面表、回归范围、经验层提示。
- 下游（testcase-generate / code-review-gate）直接消费本输出。
