# E2E / 接口验证通用模式

E2E 验证 skill 运行时加载。浏览器自动化（如 Playwright MCP）驱动真实页面 + 关键接口断言。

## 一、前置：环境与账号

E2E 前必须拿到（未提供则在部署闸口向 QA 索取）：

| 项 | 说明 |
|----|------|
| 环境 URL | 提测前端地址（若多租户，租户域名决定租户上下文请求头） |
| 租户 | 要在哪个租户下测；至少 1 个，多租户问题需 2 个 |
| 账号 | 用户名/密码/角色；权限相关需多角色账号 |
| 网络可达 | 当前会话需能访问该测试环境（内网/VPN） |
| 测试数据 | 前置数据（`<业务模块>` 相关记录）：已有则给定位方式，否则现造 |

## 二、登录模式

1. 打开环境 URL（`browser_navigate` 等价操作）。
2. 抓取页面结构（`browser_snapshot`）→ 定位登录表单。
3. 填账号密码（`browser_fill_form` / `browser_type`）→ 提交。
4. 登录后再次抓取快照，确认进入主界面。

**验证码/短信/扫码**：无法自动过 → 暂停，请 QA 协助（手动登录后接管已登录态，或提供绕过/测试专用账号）。

## 三、执行策略

- 用例来自用例生成阶段。**挑"可自动化"子集执行**：路径明确、无人工审批/支付/CA 签署的用例。
- 每条用例：操作 → 快照/截图取证 → 断言页面元素/文案/状态。
- **接口验证**：抓关键接口（`browser_network_requests` 等价操作），断言状态码、响应字段、业务码；核心金额类字段做与预期/`核心计算引擎`结果一致性核对。

## 四、不可（或不宜）自动化清单

- 真实支付、`<外部系统>` CA 签署、跨系统人工审批 → 只测到能自动化的前置环节，其余标注"需人工"。
- 依赖外部回调/异步到账的长流程 → 触发后标注"待人工确认结果"。
- 图形/短信验证码登录 → 需 QA 协助。

## 五、输出格式（回写工作项评论用）

```
## E2E/接口验证（环境 <url> / 租户 <tenant>）
通过 x / 失败 y / 阻塞 z（共 n 条可自动化用例）

| # | 用例 | 结果 | 证据(截图/接口) | 备注 |
|---|------|------|----------------|------|
| 1 | 登录并进入 <业务模块> 列表 | 通过 | 截图① | - |
| 2 | 跨租户数据隔离 | 失败 | 接口返回含他租户数据 | Blocker，见 review |

需人工验证：<列出不可自动化项>
```
