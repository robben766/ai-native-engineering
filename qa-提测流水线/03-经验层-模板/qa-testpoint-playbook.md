# QA 测试点打法 + 上线经验（方法模板）

> 用途：把 QA 的隐性验收打法显性化，供测试用例生成、E2E 模式库、白盒 review 规则复用。
> 萃取方法：从团队历史沉淀（工作项系统里状态=已实现的需求 + 其中 QA 的验收评论）里提取 `测试点：… --pass/--fail`、上线配置提醒、历史数据处理方式、需求澄清追问，归纳成"每类改动必测哪些点"。
> 与白盒 diff review 规则互补：本文是**黑盒验收维度**——每类改动"必测哪些点"。

> **如何从团队历史沉淀萃取测试点（方法）**
> 1. 从工作项系统拉取一段时间内状态=已实现的需求，聚焦某位/某组 QA 的验收评论。
> 2. 把评论里出现的固定动词/名词归类：反复出现的验证维度（导入/导出/权限/…）沉淀为"固定测试维度"；反复出现的上线动作沉淀为"上线 checklist"。
> 3. 把"实现了但不对"的返工，反向归纳成"验收前必问的澄清清单"。
> 4. 把"一个字段牵动多个模块"的验收范围，沉淀成"高影响面字段回归地图"。
> 5. 定期回流到测试用例生成骨架、review 规则、release checklist。

---

## 一、固定测试维度：任何"新增字段/新增配置"都要跑这一圈

一个"加个字段"的需求从来不是只测那个字段。反复出现的固定维度：

| 维度 | 说明（通用教学示例，不含真实数据） |
|------|-----------------------------------|
| **CRUD + 批量全套** | 新增字段必测：详情展示 / 列表展示 / 搜索过滤 / **导入 / 导出 / 批量更新 / 批量导入** 全部覆盖。例：某模块支持批量更新、导入、导出、查询后，每条路径都要单独确认字段透传。 |
| **导出的两种口径** | 批量更新导出 与 批量导出 是两条路径，字段可能一条 pass 一条漏。例：某字段在"批量更新导出"正常，但在"批量导出"里被置空——两条都要测。 |
| **菜单权限 + 数据权限 双检** | 功能权限（能不能看到菜单/按钮）与数据权限（能看到哪些数据行）是两层，各测一遍。例：某列表改为按"某归属维度（如按组织/机构）"控制数据权限，需分别验证菜单可见性与数据行范围。 |
| **多态业务对象分别验** | 同一功能对多类业务对象（如金额类/回传类/权限类 `<业务模块>`）分别生效时，**逐类** pass，绝不合并测。 |
| **多端分别验** | 各前端形态（如管理端 / 业务端 / 移动端）各测。例：业务端要对每一类业务对象各跑一遍。 |
| **配置生效范围** | 字段/配置是"仅当前页面 / 当前账号 / 全部账号"生效？例：某字段设置只对当前页面有效，刷新页面即恢复原样——生效范围必须先确认。 |

> **可直接抄进测试用例生成的清单**：新增业务字段 → 详情 × 列表 × 搜索 × 导入 × 导出 × 批量更新 × 批量导出 × 菜单权限 × 数据权限 ×（多态业务对象逐类）×（管理端/业务端/移动端）。

## 二、验收前先问：需求澄清清单（防"实现了但不对"）

在测之前常先 @产品 追问。归纳出几类必问，问不清就别开测：

1. **定制 vs 全量**："这是定制还是所有客户都需要的？"、"此字段是否某客户特有？"、"是否需要加配置开关，还是直接沿用该客户的实现方式" → 定制需求要确认作用域，避免污染全量。
2. **多态数据的归属规则**：一份数据同时被多个业务对象关联时，"关联到 A 还是 B，还是都关联？" → 一对多关系必须先定归属。
3. **配置生效范围**："配置是仅对当前页面、当前账号，还是所有账号？刷新后恢复，是否满足客户需求？" → 生效范围直接决定验收判准。
4. **历史数据怎么办**："建议先查线上库是否已有这种数据，有就明确老数据是否处理、由谁处理" → 提前定，别上线才发现。
5. **新增菜单/按钮权限 → 是否默认开放给管理员角色**：设计任何新增菜单或按钮（功能权限点）时，需求评审阶段**必须明确该权限上线后是否默认分配给（租户）管理员角色**。不问清就会出现"功能上了但客户点不到/找不到权限"。→ 评审结论要落成两件事：①默认分配给哪些角色；②是否需要上线后手动开启/授权。

## 三、上线 / 部署必知（这些坑功能测不出来，只能靠经验记住）

反复出现的**上线动作**，是纯白盒/黑盒都覆盖不到、必须落成清单的：

| 类型 | 通用经验（教学示例） |
|------|--------------------|
| **新增开关默认关，上线手动开** | 系统标准字段/新增配置默认关闭，上线后需手动开启。→ 上线 checklist 必列"要开哪些开关、给谁开权限"。 |
| **改底层生成逻辑 → 需触发一次保存刷新** | 若研发没有新增页面字段，而是修改了底层生成方法，则老数据不会自动刷新，需提示客户对相关记录"保存一下"以触发重新生成。 |
| **需要执行 migrate** | 提测评论里研发会点名"需要执行 migrate"，QA 要盯上线是否执行、执行是否成功。 |
| **历史数据修复走 SQL，不走功能** | 数据订正类需求建议直接通过数据库 SQL 处理。→ 验收=核对 SQL 结果，不是点页面。 |
| **一次性初始化 Job** | 初始化类需求上线跑一次即可。要确认 Job 的 `BEAN`/`CRON`/参数（如租户列表、起止年份等）填写正确，且 CRON 设为"只执行一次"。 |
| **访问地址(URL)变更要通知** | 对外访问路径变更（如去掉某前缀段）时，客户侧书签/`<外部系统>`对接要同步通知。 |
| **多仓 commit 追溯** | 一个需求常跨 3+ 个 `<业务仓>`，提测评论里记录各仓 SHA。回归和回滚都要按这组 SHA 走（与变更提取环节对齐）。 |

## 四、高影响面字段的回归地图（一个字段牵动一串模块）

"加个字段"经常贯穿整条数据链。QA 的验收范围就是现成的回归影响面。通用模式（用结构含义描述，不含真实业务）：

- **金额类字段**：从业务对象详情 → 单据 → 对账 → 结算 → 经营报表（**`<核心计算引擎>`脚本要能调取**）逐个落点，且导入/导出/批量更新各一遍。
- **状态类/标识类字段**：产品配置 → 各类业务对象状态 → 编辑/批量更新/导入导出 → **回传写入后端设置**（如`<外部系统>`回传后的落库）→ 结算筛选与列表 → 业务端展示。筛选该状态时要把关联记录一起筛出来。
- **配置校验类字段**：新增/编辑校验 → 单据编辑保存校验 → 新增单据校验。同一规则涉及几处入口就要几处都拦。

> **规则**：新增业务字段时，追问"哪些下游要调取"，重点排查 **对账 / 结算 / 报表（走`<核心计算引擎>`）/ 外部回传写入** 四个高关联落点。这条同时该进 review 规则的"跨仓配套/影响面"。

## 五、匹配 / 核对类算法的边界用例模板

"金额核对/多条聚合"类需求的通用边界模板：

1. 系统多条 vs 上传 1 条 → 系统金额**相加**后匹配；
2. 系统 1 条 vs 上传多条 → 上传金额**相加**后匹配；
3. 系统多条 vs 上传多条 → 双向各自相加后匹配；
4. 关联对象状态非"有效及下级" → 返回明确错误（如"记录未生效"）；
5. **全部通过才标记**（如结算年月），部分失败不标记（原子性）。

清洗规则的通用约定：匹配时可做归一化清洗（如中文括号→英文括号、中文标点→英文标点），但**「上传什么，展示什么」**——展示层不做规则优化。清洗只用于匹配，不改展示值。

## 六、回流建议

1. **测试用例生成**：把 §一的固定维度清单做成"新增字段"类需求的默认用例骨架；§五做成"核对/聚合"类的默认边界集。
2. **review 规则**：§三的"新增开关默认关/需 migrate/底层逻辑改动需刷新"补进"上线兼容性"检查；§四的高关联落点补进"跨仓配套/影响面"。
3. **上线 checklist（新增）**：开关手动开 + 权限分配 + migrate + 一次性 Job（参数核对）+ URL 变更通知 + 历史数据 SQL 订正——建议单独维护一份 release checklist。
4. **E2E 模式库**：多端（管理端/业务端/移动端）× 多态业务对象的矩阵化验证，可作为 E2E 断言的标准展开方式。
