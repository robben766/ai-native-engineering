# 经验层（Experience Layer）

提测验证流水线的"血肉"。引擎（`/test-intake` + skills）通用不变，**项目差异性和踩坑经验都沉淀在这里**，运行时按目标仓自动加载注入。

## 结构

```
docs/experience/
├── README.md            # 本文件
├── platform.md          # 平台级通用经验（所有仓通用的坑）
├── production-bug-patterns.md # 生产逃逸缺陷高频模式 + 漏测复盘结论 + 回流
├── qa-testpoint-playbook.md   # 从已实现需求评论萃取:测试点固定维度 + 上线/数据处理经验
├── review-rules.md      # 代码 review 检查清单 + 门禁自动裁决规则
├── e2e-patterns.md      # E2E 通用模式：登录、验证码、断言、测试数据、不可自动化清单
├── oracles.md           # 判准库：某功能"什么结果才算对"，用例生成定预期 / E2E 断言时消费并沉淀
├── projects/
│   ├── _template.md     # 新建项目经验的模板
│   ├── <业务仓A>.md
│   ├── <业务仓B>.md
│   └── <模块>.md
└── scenarios/           # 按业务领域沉淀的测试场景/操作手册（QA 历史经验）
    ├── <业务模块-金额类>.md   # 例：结算/报表类测试场景（示例）
    ├── <外部系统对接>.md      # 例：财务系统对接测试流程（示例）
    └── <业务模块-回传类>.md   # 例：单据配置与回传测试手册（示例）
```

## 场景经验索引（scenarios/）

| 文档 | 内容 | 关联业务领域 |
|------|------|-------------|
| [<业务模块-金额类>](scenarios/<业务模块-金额类>.md) | 报表新增流程、金额结算、数据准确性与超时坑（示例） | `<领域文档>/<金额类模块>/` |
| [<外部系统对接>](scenarios/<外部系统对接>.md) | 业务数据同步到 <外部系统> 的配置与测试要点（示例） | `<领域文档>/<财务领域>/integration/` |
| [<业务模块-回传类>](scenarios/<业务模块-回传类>.md) | <外部系统> 配置、单据生成与影像回传测试（示例） | `<领域文档>/<业务领域>/` |

> 说明：场景文档按业务领域分类沉淀。建议区分**金额类**（结算/对账/报表，重点验数据精度与一致性）、**回传类**（对接外部系统、影像/文件回传）、**权限类**（多租户、角色可见范围）等结构，便于加载时按领域命中。

## 加载规则（引擎运行时）

某个提测的目标仓确定后，流水线加载：

```
platform.md  +  projects/<目标仓>.md  +  当前阶段规则(review-rules.md / e2e-patterns.md)
```

## 自我迭代（核心）

每次跑完，或过程中发现问题，把经验沉淀回来：

- **漏检了某类问题** → 加进 `review-rules.md` 对应检查项。
- **某仓有特殊约定/坑** → 加进 `projects/<仓>.md`。
- **某种 E2E 场景的处理方式**（如某验证码怎么过、某测试数据怎么造）→ 加进 `e2e-patterns.md`。
- **平台级通用坑** → 加进 `platform.md`。

追加方式：过程中一句"这个记下来"，就 append 到对应文件（**只改本地，不 commit**）。

> 写经验的标准：**具体、可操作、带证据**。不写"要注意多租户"，而写"`XXController` 的 list 接口未带租户过滤，跨租户会串数据，见 <工作项系统> 编号（示例）"。

## 附：判准库（oracles.md）写法约定

判准（oracle）回答"某功能什么结果才算对"，供用例生成定预期、E2E 断言时消费。

**什么时候用**：一个功能存在明确的"正确/错误"边界，且容易被误判或漏测时（金额精度、幂等、权限可见范围、边界取值、并发结果、兼容性）。

**结构**（每条判准建议包含）：

```
- 功能/接口：<模块>/<接口路径>
- 前置条件：<租户/角色/数据状态>
- 预期结果：<明确的、可断言的正确输出>
- 反例：<常见的错误表现>
- 置信度：高 / 中 / 低（低置信度需人工复核）
- 证据：<工作项系统编号（示例） / 复盘结论 / 代码位置>
```

**置信度**：高=有明确规则或复盘背书可直接门禁；中=需结合上下文；低=仅提示，人工裁决。

## 附：代码 review 检查清单通用类目

沉淀到 `review-rules.md`，作为门禁裁决输入。通用类目（任何业务都适用）：

- **多租户**：list/detail/export 接口是否带租户过滤，跨租户是否串数据。
- **权限**：角色可见范围、按钮级/数据级权限、越权访问。
- **金额精度**：小数位、四舍五入方式、单位换算、汇总口径一致性。
- **幂等**：重复提交、回调重放、定时任务重跑是否产生脏数据。
- **边界**：空值、极值、分页边界、日期跨界。
- **并发**：并发写、库存/额度扣减、乐观锁/悲观锁。
- **SQL**：全表扫描、缺索引、N+1、大事务、注入。
- **兼容性**：接口出入参变更、DB migrate 向前/向后兼容、老数据。

## 附：门禁自动裁决机制

review 与 E2E 阶段产出问题清单后，按规则自动裁决是否放行：

- **阻断（Block）**：命中高置信度判准的正确性问题、多租户/越权/金额错误等高危类目 → 不放行。
- **警告（Warn）**：中置信度问题、非核心路径缺陷 → 放行但记录。
- **通过（Pass）**：无命中或仅低置信度提示。
- **人工覆盖**：任何自动裁决均可由人工复核覆盖，覆盖理由回写工作项。
