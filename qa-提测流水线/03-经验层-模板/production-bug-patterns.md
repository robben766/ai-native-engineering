# 生产环境缺陷模式沉淀（模板）

> **本文件是空模板**。用途：把「逃逸到生产的缺陷」从工作项系统拉出来、聚类、萃取高频模式，再回流到经验层（评审规则 / 判准 / 用例生成）。
> 按下方方法执行一遍，用你自己项目的真实数据把表填满即可。所有 `<占位符>` 与「（示例）」均需替换或删除。

---

## 〇、为什么单独沉淀"生产缺陷"

生产缺陷 = **测试没拦住的那部分**，是回归用例与门禁最该对齐的靶子。它与普通缺陷的分布通常显著不同：严重级别更高、客户侧一暴露即高优先级。把这批缺陷单独聚类，能直接告诉你"钱应该压在哪些回归用例上"。

**建议对齐的下游产物**：评审规则库（`review-rules.md`）、判准库（`oracles.md`）、用例生成器（`testcase-generate`）。

---

## 一、方法：从工作项系统拉数据 → 聚类 → 萃取 → 回流

### 步骤 1：拉取生产缺陷样本

从「工作项系统」按下列条件拉一批缺陷：

- **筛选维度**：`发现阶段 = 生产环境`（字段名各系统不同，找"发现阶段/originphase/环境"类字段）。
- **排序**：按创建时间倒序。
- **样本量**：取最近 N 条（建议 100~300，兼顾时效与统计意义）。
- **区间**：记录数据区间起止，便于复盘时对齐。
- **落盘**：把原始拉取结果存成快照文件（如 `prodbugs.json`），刷新时重跑同一流程，保证可复现。

> 拉取方式：优先用工作项系统的 API / MCP；无接口时用导出。记录 `workspace/项目 id`、筛选条件、limit，写进本文件顶部的「来源」块。

### 步骤 2：先看两个反差（严重度分布）

拉完先做一次"生产 vs 全量"的对照，确认这批数据确实更严重、更值得投入：

| 维度 | 生产缺陷（本批 N 条） | 全量对照 | 含义 |
|------|------:|------:|------|
| serious+fatal 占比 | （示例）% | （示例）% | 逃逸到生产的多为真功能坏 / 资损 / 数据丢 |
| high+urgent 占比 | （示例）% | （示例）% | 客户侧暴露即高优先级 |

### 步骤 3：按「线上问题类型」分桶

若工作项系统有"线上问题类型"字段，按它分桶；没有则按标题人工打标。典型桶：

| 类型 | 数量 | 解读方向 |
|------|-----:|------|
| 新缺陷 | （示例） | 近期需求引入的新代码坏 → 新功能回归是主战场 |
| 拒绝BUG | （示例） | 被拒的生产单 → 见 §三「误报治理」 |
| 外部缺陷 | （示例） | `<外部系统>` 回传 / 对接方问题 |
| 需求缺口 | （示例） | 生产暴露的需求缺口，不是 bug，应转需求 |
| 历史缺陷 | （示例） | 老功能长期带病 |
| 设计 / 性能 / 部署配置 / 兼容 | （示例） | |

### 步骤 4：画「雷区地图」（按领域聚类）

按业务领域聚类，找出缺陷密度最高的模块。用你项目的领域切分填表（保留"金额类 / 回传类 / 权限类模块"这种结构含义，不必写真实模块名）：

| 领域 / `<业务模块>` | 数量 | 占比 |
|------|-----:|-----:|
| `<回传 / 数据接入类模块>` | （示例） | （示例）% |
| `<金额 / 对账 / 结算类模块>` | （示例） | （示例）% |
| `<移动端 / 展业类模块>` | （示例） | （示例）% |
| `<机构 / 人员 / 权限类模块>` | （示例） | （示例）% |
| 其他 | — | |

再按**缺陷类型标签**做一次正交聚类（标题多标签）：金额/费率、导入导出、数据不一致、报错500、显示为空、附件/影像、外部回传识别、签署 等，各计数。

### 步骤 5：萃取高频模式（核心产出）

对每个高频簇，用统一四段式写成一条可复用模式（见 §三 表头）：**现象 → 推断根因 → 复用规则 → 证据**。判据：一个模式至少应由多条同类缺陷支撑，且能推导出一条可加进评审规则 / 判准的检查项。

### 步骤 6：回流经验层

把萃取出的规则写进下游产物（见 §五「回流动作」），并记录落地状态。下次真实提测即自动应用。

## 二、聚类与萃取的判准（oracle）约定

萃取模式时，凡涉及"对不对"的判断，都用统一的判准格式沉淀到 `oracles.md`。判准三要素：

- **什么时候用**：触发场景（命中哪些模块 / 数据形态）。
- **结构**：可断言的等式或状态约束，例如「明细合计 = 主表金额」「列表金额 = 详情金额」「多数据源三处状态一致」。
- **置信度**：强制断言 / 建议核对 / 采样抽查。

判准应当是**机器可断言**的，才能被用例生成与门禁复用。

## 三、高频生产模式表（空表头 + 字段说明）

> 这是模板的核心交付物。每萃取出一个高频簇，填一行 / 一块。下面先给**字段说明**，再给**空表头**，最后给一个**教学性通用示例**（不含任何真实数据）。

### 字段说明

| 字段 | 含义 | 填写要求 |
|------|------|---------|
| 模式编号 | P1、P2… | 按危害/频次排序，最高危在前 |
| 模式名 | 一句话概括根因类别 | 用通用语言，不写真实模块/客户名 |
| 现象 | 该簇缺陷的可观察表现 | 列 3~5 条典型现象，脱敏 |
| 推断根因 | 从现象反推的共性根因 | 标注"推断"，未证实不要写死 |
| 复用规则 | 可加进 `review-rules.md` 的检查项 | 必须可执行、可断言 |
| 判准(oracle) | 可加进 `oracles.md` 的强制断言 | 见 §二 三要素 |
| 回归造数 | 该模式对应必造的边界数据 | 指导 `testcase-generate` |
| 证据 | 支撑该模式的缺陷 | **只留编号/计数，删除客户名与真实标题** |

### 空表头（复制此块，逐个模式填写）

```
### P<n>. <模式名>
- 现象：<现象1>；<现象2>；<现象3>。
- 推断根因：<共性根因，标注为推断>。
- 复用规则：<可加进 review-rules 的检查项>。
- 判准(oracle)：<可加进 oracles 的强制断言>（触发场景 / 结构 / 置信度）。
- 回归造数：<必造的边界数据形态>。
- 证据：<缺陷计数或编号，脱敏>。
```

### 通用示例（教学用，非真实数据）

以下示例仅示范"如何写一条模式"，请勿保留到你的正式文档，用你自己的数据替换。

**示例 A — 外部回传唯一键错导致数据丢失（回传类模块通用陷阱）**
- 现象：同一实体多条明细只落库一条；不同子类被合并成同一条；关键字段为空却静默入库。
- 推断根因（推断）：落库唯一键只用了粗粒度主键，未含业务子维度；识别/去重合并覆盖了应保留的多条；空关键字段无拦截。
- 复用规则：`<外部系统>` 回传唯一键必含业务子维度；同键多条要么全留要么显式合并留痕；关键字段解析失败要**拦截+告警**不静默。
- 判准(oracle)：回传入库条数 = 源报文有效明细条数（强制断言）。
- 回归造数：一实体多子类 / 同子类多值 / 关键字段缺失 三种边界报文。
- 证据：（示例）N 条。

**示例 B — 数据权限过滤路径不全（权限类模块通用陷阱）**
- 现象：搜索返回越权数据；导出比列表多带出无权数据；异步任务不带权限过滤。
- 推断根因（推断）：数据权限只加在列表主查询，搜索/导出/异步四条路径口径不统一。
- 复用规则：数据权限过滤必须覆盖**搜索、列表、导出、异步任务**四条路径。
- 判准(oracle)：同一用户在四条路径可见数据集合一致（强制断言）。
- 回归造数：跨机构 / 跨层级 / 未勾选下级 三种权限组合。
- 证据：（示例）N 条。

**示例 C — 金额精度与明细≠汇总（金额类模块通用陷阱）**
- 现象：结算金额列表≠详情；合计人数与明细人数不符；保存/导入产生重复且不能删。
- 推断根因（推断）：汇总与明细两套算法；方案变更未级联刷新；导入去重缺失。
- 复用规则：金额一律走高精度类型（如 BigDecimal / 「核心计算引擎」）；方案变更级联刷新下游记录；导入必去重。
- 判准(oracle)：明细合计 = 主表金额、列表金额 = 详情金额（强制断言）。
- 回归造数：多币种/多费率枚举、方案变更前后、重复导入。
- 证据：（示例）N 条。

## 四、比缺陷本身更重要的三条过程结论（复盘模板）

聚类完成后，除了模式表，还要输出"过程结论"——它决定门禁与流程怎么改。以下三条是通用复盘视角，填入你的数据：

### 结论1：新缺陷占比 → 决定回归资源投向

统计"新缺陷 / 总样本"占比。若过半，说明生产逃逸主要来自**近期需求引入的新代码**而非老功能腐化 → 门禁与回归用例应压在**本次变更相关模块 + 其直接下游**，而非平均撒网。配合变更影响面反查（`mr-change-extract` 类工具）。

### 结论2：拒绝BUG 占比 → intake 分级是否有漏

统计"拒绝BUG / 总样本"占比。被拒单常见为：需求/客户定制 by-design、权限 by-design、无法复现/环境数据特有。
→ 改进：生产问题登记时先过一道**分流**（需求缺口 / 客户定制 / 环境数据 / 真缺陷），减少研发被无效单打断；真需求缺口应转需求而非当 bug 修。

### 结论3：漏测归因字段是否填全 → 能否算出真实漏测率

关键盲区：若"测试环境是否可复现"（线上BUG专属字段）大面积为空，就**无法区分**：
- **测试环境可复现** → 纯漏测，直接反哺用例库；
- **仅生产可复现** → 环境/客户数据/回传特有，反哺测试数据构造与环境治理。
→ 建议：serious 以上生产缺陷，关闭前**强制**填"测试环境是否可复现" + "线上问题类型"。否则复盘只能停在标题推断，度量体系缺最关键的漏测归因维度。

## 五、回流动作（可直接落地的清单）

把萃取结果写回经验层，并记录落地状态（✅已落地 / 🔺待推动）：

1. **`review-rules.md` 新增检查项**（每条来自一个高频模式 P<n>）：
   - `<回传唯一键>`：外部回传唯一键是否含业务子维度；同键多条合并规则；关键字段解析失败是否拦截。
   - `<数据权限四路径>`：搜索/列表/导出/异步任务是否都带数据权限过滤。
   - `<金额一致性>`：明细合计=主表、列表=详情，作为强制断言。
   - `<外部回调幂等>`：回调落库失败/重复回调/覆盖的幂等与补偿。
2. **`oracles.md` 补判准**：把 §二/§三 各模式的判准（多数据源一致、明细=汇总、枚举匹配等）沉淀为强制断言。
3. **`testcase-generate` 回归权重**：命中高频模块（`<回传>`/`<金额结算>`/`<权限>`/`<附件>`/`<外部对接>`）的提测，默认加挂对应生产高频用例集。
4. **流程（需团队推动）**：推动 serious 生产缺陷强制填"测试环境可复现" + "线上问题类型"，下次复盘即可从"标题推断"升级为"统计归因"。

## 六、门禁裁决与自我迭代（机制说明）

- **门禁裁决**：上述复用规则/判准接入提测门禁后，命中即产出问题清单 + 风险分级 + 自动裁决（阻断/放行），支持人工覆盖并留痕。
- **自我迭代**：每次生产缺陷复盘都重跑本流程（步骤1~6），新模式增量并入模式表与规则库，旧模式若长期零命中可降权或归档。
- **经验层加载规则**：下游工具（评审/用例生成）在对应模块任务时按需加载本文件与 `review-rules.md`/`oracles.md`，不平均全量加载。
